# /skills-as-rules

Goal: convert the project's **existing AI rules, conventions, and tribal
knowledge** into proper [Agent Skills](https://docs.anthropic.com/en/docs/build-with-claude/agent-skills)
that live under `ai-specs/skills/<name>/SKILL.md` and get auto-invoked through
the `AGENTS.md` table.

This is a one-time (or rarely-repeated) bootstrap. Run it after `specs-ai init`
on an established codebase that already has implicit conventions written down
somewhere — Cursor rules, Copilot instructions, contributing guides, hand-rolled
`AGENTS.md` content, etc.

---

## What you (the agent) must do

### 1. Discover existing rule sources

Look for any of these — they encode what we want to migrate:

- `.cursor/rules/*.mdc`, `.cursorrules`
- `.github/copilot-instructions.md`
- `CLAUDE.md`, `GEMINI.md`, `AGENTS.md` (only the parts NOT auto-generated by `specs-ai sync`)
- `CONTRIBUTING.md`, `STYLEGUIDE.md`, `ARCHITECTURE.md`, `docs/conventions/**`
- Per-area READMEs that describe coding patterns
- ADRs (`docs/adr/**`, `docs/decisions/**`)
- Inline comments at the top of foundational modules ("// don't do X here, do Y")

Read enough of each to understand the **distinct concerns** they cover.

### 2. Group by domain, not by source

Group the rules by **what slice of the codebase they govern**, not by the file
they came from. Examples of good slicings:

- `<project>-api` — backend conventions (framework, layering, testing)
- `<project>-frontend` — UI conventions (component style, state mgmt, testing)
- `<project>-commit` — git workflow (branches, commits, PRs)
- `<project>-docker` — container / dependency management
- `<project>-<bounded-context>` — anything else with its own rules

A skill should be **invoke-able** — a clear answer to "when should the agent
load this?". If you can't write a one-line trigger for it, the slicing is
wrong.

### 3. For each group, generate a skill

Use the bundled `skill-creator` skill as your authoring guide:

```
Read ai-specs/skills/skill-creator/SKILL.md
```

Then for each group, create `ai-specs/skills/<name>/SKILL.md` with valid
frontmatter:

```yaml
---
name: <name>
description: >
  <one-paragraph "what this is + when to invoke it">.
  Trigger: <comma-separated list of situations that should auto-load this skill>.
auto_invoke:
  - "When working on <concrete trigger A>"
  - "When working on <concrete trigger B>"
---
```

Body: distill the source rules into prescriptive, actionable guidance. Drop
duplication. Drop anything that's just tribal lore without operational impact.
Keep examples short and concrete.

### 4. Refresh the index

Once all skills exist, run the bundled `skill-sync` skill to refresh the
Auto-invoke table in `AGENTS.md`:

```
Read ai-specs/skills/skill-sync/SKILL.md
```

Then execute its `assets/sync.sh`. Verify the new skills show up in the
`### Auto-invoke Skills` table at the project root.

### 5. Stop. Confirm with the user before deleting source files.

Do **NOT** delete `.cursor/rules/*.mdc`, `CONTRIBUTING.md`, etc. on your own.
List what was migrated, ask the user which sources are now redundant, and
let them remove or shrink them.

---

## Output expectations

When you finish, report back:

1. A list of skills created (`ai-specs/skills/<name>/`)
2. The source files each skill drew from
3. Anything you intentionally **did not** migrate, and why
4. A reminder for the user to commit `ai-specs/skills/` and the regenerated `AGENTS.md`

---

## Anti-patterns

- ❌ One mega-skill that covers everything. Skills should be loadable
  independently.
- ❌ Copy-pasting raw `.mdc` files into `SKILL.md`. Re-write for the new
  voice and structure.
- ❌ Skipping the `auto_invoke` block. Without it, the skill won't show up in
  the Auto-invoke table and agents won't load it on the right triggers.
- ❌ Touching generated content in `AGENTS.md` by hand. Always let
  `specs-ai sync` and `skill-sync` regenerate it.
