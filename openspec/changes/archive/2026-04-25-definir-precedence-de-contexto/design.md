## Context

The project currently has several context carriers with different authority levels: repository documentation, project-local skills, vendored or bundled packs, handoff notes, runtime/session memory, and proposed agent output. The manifest contract explicitly deferred precedence rules, so this change introduces a small standalone capability instead of expanding `ai-specs.toml` semantics.

The implementation should stay documentation-first. `AGENTS.md` is generated by `lib/_internal/agents-md-render.py`, so any agent-facing reference must come from the renderer or a source that `ai-specs sync` regenerates, not from hand-editing `AGENTS.md`.

Simple discovery flow:

```text
docs/ai/context-precedence.md
        |
        | referenced by
        v
lib/_internal/agents-md-render.py
        |
        | ai-specs sync / sync-agent
        v
AGENTS.md in root and declared targets
        |
        v
agent reads precedence rule during startup/context building
```

## Goals / Non-Goals

**Goals:**

- Publish a canonical precedence order for project context conflicts.
- Define each source class in plain language so agents and humans can audit a decision.
- Provide concrete examples for common conflicts between docs, skills, packs, handoffs, memory, and proposed content.
- Make the rule discoverable from generated agent instructions without hand-editing generated files.
- Cover docs and generated output with focused tests.

**Non-Goals:**

- No runtime context router or automated merge engine.
- No new `[memory]`, `[precedence]`, or pack-related manifest fields.
- No changes to OpenSpec artifact semantics, Trello workflow, or runtime memory storage.
- No attempt to decide every future precedence nuance beyond the MVP source order.

## Decisions

| Decision | Choice | Alternatives considered | Rationale |
|---|---|---|---|
| Authority model | Use one total order: canonical docs > project skills > packs > handoffs > session memory > proposed context | Per-topic precedence matrix or runtime-scored freshness | The card asks for a simple auditable MVP. A total order is easy to explain and test. |
| Canonical location | Add `docs/ai/context-precedence.md` | Add only to README or only to AGENTS.md | `docs/ai` is the intended canonical documentation space, while generated AGENTS can link to it. |
| Generated reference | Update `lib/_internal/agents-md-render.py` so regenerated `AGENTS.md` points to the doc | Hand-edit `AGENTS.md`; duplicate the full rule in every generated target | Generated files must not be hand-edited. A renderer reference keeps root and subrepo outputs aligned. |
| README role | Add a short pointer in `README.md`, not the full rule | Duplicate the full precedence doc in README | README should help discovery without becoming a second source of truth. |
| Testing | Add doc assertions plus renderer/sync assertions | Rely on manual review | The change is documentation-first, so tests should verify the exact order and generated discoverability. |

## Risks / Trade-offs

| Risk | Mitigation |
|---|---|
| Agents treat the docs rule as a runtime guarantee | Explicitly state that the MVP is a human/agent decision rule, not an automated router. |
| README, docs, and AGENTS drift | Keep the full normative rule in `docs/ai/context-precedence.md`; use README and AGENTS as references only; test exact references. |
| The total order is too coarse for future pack/runtime behavior | Mark it as MVP and leave room for a future Context Router change to refine or automate it. |
| Generated AGENTS reference breaks subrepo relative links | Use a repo-relative path that remains valid after root artifacts are mirrored to declared targets. |

## Migration Plan

1. Add `docs/ai/context-precedence.md` with the normative order, source definitions, conflict examples, and audit checklist.
2. Add a short README pointer to the new canonical doc.
3. Update `lib/_internal/agents-md-render.py` to render a compact context precedence section in generated `AGENTS.md`.
4. Add tests that assert the canonical order appears in the doc and that generated AGENTS output references it.
5. Run `ai-specs sync` so `AGENTS.md` reflects the renderer change.
6. Run `./tests/run.sh` and `./tests/validate.sh`.

Rollback: revert the doc, README pointer, renderer addition, generated `AGENTS.md`, and related tests. No data migration is required because no runtime state or manifest schema changes are introduced.
