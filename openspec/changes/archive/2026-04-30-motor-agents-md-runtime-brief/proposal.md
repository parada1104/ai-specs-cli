## Why

`AGENTS.md` currently serves two incompatible roles: it is both the project's runtime operational brief (MCPs, safety rules, context sources) and an exhaustive, auto-generated skill registry with Auto-invoke tables. This bloats the context window with duplicated information that agents can already discover from the skills directory, and it blocks recipe `trello-mcp-workflow` (#62) from having a clean runtime brief. We need to split these concerns so `AGENTS.md` becomes a concise, human-meaningful runtime context document while skills remain discoverable through a separate, idempotent generated artifact.

## What Changes

- **Redesign `lib/_internal/agents-md-render.py`** to generate a runtime brief from `ai-specs.toml` instead of a skill registry. The output includes project identity, enabled runtimes, MCP servers (without secrets), recipes/bindings/capabilities when present, safety rules, and context source precedence.
- **Remove exhaustive skills table and `Auto-invoke Skills` section** from generated `AGENTS.md`. Skills are still discoverable via the filesystem and the new registry artifact.
- **Modify `skill-sync`** (`ai-specs/skills/skill-sync/assets/sync.sh`) to validate skill metadata and generate a separate registry file (e.g., `ai-specs/.skill-registry.md` or equivalent) without rewriting `AGENTS.md`. The existing runtime-brief marker check becomes the permanent behavior.
- **Generate an idempotent skill/capability registry artifact** outside `AGENTS.md` that contains the canonical skill index and Auto-invoke mappings. This artifact is treated as a derived output (regenerated by sync, not hand-edited).
- **Include configured MCPs in the brief** with redacted secrets (env variable references shown, never literal values).
- **Include recipes, bindings, and capabilities** in the brief when they exist in the manifest, so agents know what operational bundles are active.
- **Add idempotency tests**: running `ai-specs sync` twice with the same `ai-specs.toml` must produce byte-identical `AGENTS.md` and registry artifacts.
- **Update documentation and README** to reflect the new `AGENTS.md` purpose and the separate registry artifact.

## Capabilities

### New Capabilities
- `agents-md-runtime-brief`: Defines the content contract for `AGENTS.md` as a runtime brief derived from `ai-specs.toml` — project metadata, enabled agents, MCPs, active recipes/bindings, safety rules, context sources, and workflow rules. Does NOT include skill catalogs or Auto-invoke tables.
- `skill-registry-artifact`: Defines the separate, generated skill/capability registry artifact (e.g., `ai-specs/.skill-registry.md` or equivalent) that contains the canonical skill index and Auto-invoke mappings. Treated as a derived artifact: regenerated by sync, never hand-edited.

### Modified Capabilities
- `recipe-sync-materialization`: Delta needed. The "derived artifacts (AGENTS.md, agent configs) SHALL reflect new skills and commands" scenario must be updated to state that skills are reflected in the separate registry artifact, not in `AGENTS.md`.
- `skill-frontmatter-contract`: Delta needed. The requirement "generated AGENTS.md MUST include one Auto-invoke row per trigger for matching scopes" must be updated to state that Auto-invoke rows are emitted into the separate registry artifact, not into `AGENTS.md`.

## Impact

- `lib/_internal/agents-md-render.py` — complete rewrite of output format.
- `ai-specs/skills/skill-sync/assets/sync.sh` — output target changes from `AGENTS.md` to registry artifact; runtime-brief skip becomes permanent.
- `ai-specs.toml` contract documentation — may need updates to document new derived artifacts.
- `AGENTS.md` in this repo — will be regenerated and must be reviewed to ensure the manual runtime-brief content is correctly captured in the new template.
- Test suite — new idempotency tests and updates to existing sync tests.
- README/docs — updated descriptions of `AGENTS.md` purpose and registry artifact.
